## Frustum display
For every visible camera object: wireframe pyramid from the camera origin
through the 4 corners of a virtual image plane at ~1.5 units along
objectForward (use cameraFovY and a 16:9 aspect for the plane), plus an
"up" triangle above the top edge (Blender's marker). LINES via a small
per-camera VertexArray cached on (focalLength) — model matrix positions it,
so only focalLength changes rebuild. Draw after the grid, before outlines.
Active camera's frustum in the selection orange when selected, light grey
otherwise (same colors as iconPass).

## View-through-camera (Numpad0)
- Extract PURE helpers (new file or cameraFrustumPass.ts, your call — say
  where in Result): `cameraViewMatrix(obj)` = inverse of the camera's
  translation+rotation (IGNORE scale), `cameraProjMatrix(data, aspect)` =
  Mat4.perspective(cameraFovY(data), aspect, near, far).
- Renderer gets `cameraViewId: number | null` (public field). In render() and
  pick(): if set and scene.get(id) is a visible camera, use the helpers for
  view/proj instead of the OrbitCamera; else fall through (and null the field
  if the object is gone). Skip drawing the frustum + icon of the camera being
  looked through. Everything downstream (picking, gizmo, edit overlays)
  already takes view/proj from those locals — verify by reading render().
- InputManager: Numpad0 (e.code === 'Numpad0', no modifiers, object OR edit
  mode) toggles renderer.cameraViewId between null and
  scene.activeCameraId (status text 'View: Camera' / 'View: User').
  Orbit/pan/zoom input while in camera view exits it first (one-liner where
  orbit starts — find the wheel + MMB handlers).
- e2e escape hatch: the field is public — tests may set it directly.

# UR4-1 — Continuous grab: pointer lock, unclamped modal pointer, Shift precision

## Goal
Modal transforms (G/R/S, edge slide, …) currently die at the canvas/window edge —
the mouse leaves and events stop. Blender-style fix: while a modal transform is
active, capture the pointer (Pointer Lock API), accumulate raw movement deltas into
a VIRTUAL pointer position with no bounds, and feed THAT to the operator. Holding
**Shift** scales deltas to 0.1 (precision mode) with no jump on press/release.

## Files
- NEW `src/input/modalPointer.ts` + unit tests — the pure accumulator:
  `class ModalPointer { begin(x,y); move(dx,dy, precise: boolean): {x,y}; readonly pos }`
  pos = begin pos + Σ(delta × (precise ? 0.1 : 1)). No DOM in this file.
- EDIT `src/input/InputManager.ts` — wire it (details below).
- EDIT `src/core/operator/Operator.ts` — optional
  `readonly continuousGrab?: boolean` on the Operator interface.
- EDIT the operator classes that opt in (add `readonly continuousGrab = true`):
  TranslateOperator, RotateOperator, ScaleOperator, their edit-mode variants,
  EdgeSlideOperator. Do NOT opt in BoxSelect, Knife, LoopCut, sculpt (they need
  the real cursor).
- NEW small CSS/DOM: a software cursor (crosshair div inside `#viewport-wrap`)
  shown while pointer-locked, positioned at the virtual pos WRAPPED into the
  canvas rect (modulo), Blender-style. Hidden otherwise.

## Mechanism (InputManager)
- `startOperator`: if `op.continuousGrab` and the canvas supports it, call
  `canvas.requestPointerLock()` and `modalPointer.begin(this.pointer.x, .y)`.
- While locked, `pointermove` uses `e.movementX/Y` →
  `modalPointer.move(dx, dy, e.shiftKey)` → the result becomes the PointerState
  passed to `op.onPointerMove` (unclamped — operators already handle arbitrary
  coords; rotate's pivot math works with out-of-canvas positions).
- While an op with continuousGrab is active but lock FAILED or is unavailable
  (headless e2e!), fall back to the same accumulator fed by real-event deltas
  (difference from last event position, scaled by shift) — precision mode then
  works everywhere even though edge-of-screen capture doesn't.
- **Escape under pointer lock:** the browser exits lock and SWALLOWS the Esc
  keydown. Listen to `pointerlockchange`: if lock is lost while a continuousGrab
  op is active and we didn't end it ourselves → cancel the operator (this keeps
  Esc = cancel). Confirm/cancel paths call `document.exitPointerLock()`.
- Modifier-only keydown/keyup (Shift) must not disturb operators; precision is
  read from the move events' shiftKey, so no extra key handling.
- Non-modal pointer handling (camera orbit, click select, gizmo) unchanged.

## Out of scope
- No changes to operator math or numeric input.
- No pointer lock outside modal operators.
- Rotate/scale UX beyond feeding them the virtual pointer.

## Acceptance criteria
- `npx tsc --noEmit` clean; `npm test` green; ModalPointer unit tests cover
  precision transitions mid-gesture (no position jump) and unclamped accumulation.
- e2e (new `e2e/modal-pointer.mjs`, unique E2E_PORT): headless has no pointer
  lock, so test the FALLBACK path: (1) G then dispatch moves — object moves;
  (2) same distance with shiftKey → ~0.1× the displacement; (3) shift pressed
  mid-move → no jump (position continuous); (4) moves far outside the canvas
  rect keep applying (dispatch coords beyond the canvas bounds; virtual pointer
  keeps integrating). Escape still cancels.
- Manual-check note for the architect in your report: anything you could not
  verify headless (real pointer lock, software cursor wrap).
- If the spec is ambiguous or wrong, stop and report rather than improvising.

# P7-4 — Grid snapping

Read `tasks/P4-CONVENTIONS.md` first.

## Goal
**Shift+Tab** toggles snap mode (both modes; topbar shows a 🧲 chip state,
data-action="snap-toggle", clickable). While snapping is ON, modal MOVE
operators (object G/gizmo drag AND edit-mode G) round each moved point's
world position to 0.5-unit increments (Blender's "increment" snap, ours fixed
at 0.5). Holding **Ctrl** during the modal INVERTS the current snap state for
as long as it's held (snap-off → Ctrl snaps; snap-on → Ctrl disables).

## Files
- NEW `src/core/snap.ts` — module-level `snapState = { enabled }` + pure
  `snapVec(v, step)` + test
- `src/tools/translate.ts` + `src/tools/editTransform.ts` — apply snapping in
  the translate paths (snap the RESULT positions, not the delta; Ctrl detection
  via the modal onKey/keyup — InputManager must forward keyup for this, add the
  narrowest possible hook)
- `src/input/InputManager.ts` — Shift+Tab (before the plain-Tab edit toggle!),
  keyup forwarding hook
- `src/ui/topbar.ts` — magnet chip
- `e2e/edit.mjs` or `e2e/ship.mjs` — checks (your choice, say which)

## Acceptance criteria
1. Unit: snapVec rounds each component to the step (1.24→1.0, 1.26→1.5, negatives).
2. e2e: snap on via Shift+Tab (chip reflects it), G-move the cube → confirmed
   position components are exact multiples of 0.5; snap off → a drag lands on
   a non-multiple (move by an odd pixel amount); plain Tab still toggles edit
   mode (regression check).
3. All four e2e suites + npm run build + npm test pass.

## Result
Implemented grid ("increment") snapping fixed at a 0.5-unit step.

- **NEW `src/core/snap.ts`** — module-level `snapState = { enabled }`, `SNAP_STEP = 0.5`, pure `snapVec(v, step)` (component-wise round to the nearest multiple; non-positive step is a no-op), and `snapActive(ctrlHeld)` = `snapState.enabled !== ctrlHeld` (Ctrl inverts). Unit test `src/core/snap.test.ts` covers 1.24→1.0, 1.26→1.5, negatives, exact multiples, the no-op step, and the XOR inversion.
- **`src/tools/translate.ts`** (object G / gizmo drag) and **`src/tools/editTransform.ts`** (edit-mode G) — snap the RESULT world position (not the delta). Each op tracks a `ctrlHeld` flag toggled by `onKey('Control')` / new `onKeyUp('Control')` and re-applies live. Edit-mode keeps the original local math on the non-snap path (byte-identical to before); the snap path snaps world then converts back through the inverse matrix.
- **`src/core/operator/Operator.ts`** — added optional `onKeyUp?(ctx, key)` to the Operator interface (narrowest hook).
- **`src/input/InputManager.ts`** — added a window `keyup` listener that forwards only to `activeOp.onKeyUp`; added a **Shift+Tab** grid-snap toggle placed BEFORE the plain-Tab edit toggle so the modifier form wins.
- **`src/ui/topbar.ts`** + **`src/ui/theme.css`** — clickable 🧲 magnet chip (`data-action="snap-toggle"`, `aria-pressed`, `.topbar-btn-on` highlight) mirroring `snapState.enabled`; toggles on click, reflects Shift+Tab every frame.
- **e2e checks in `e2e/ship.mjs`** — snap chip starts off; Shift+Tab flips it on (chip reflects via `aria-pressed`); plain Tab still enters/exits edit mode and leaves the chip untouched (regression); a snapped G-move confirms to exact 0.5 multiples; snap off → an odd-pixel drag lands off-grid.

**Verification:** 231 unit tests pass (incl. the new 8-case snap.test.ts), `npm run build` clean, all 4 e2e suites pass (smoke, edit, ship, workspace). No deviations from the spec.

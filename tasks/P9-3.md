## The panel
After AddMenu adds a primitive, mount an "Add <Name>" collapsed-card panel at
the viewport's bottom-left (Blender's spot). One input row per param (number/
int/checkbox). Changing a value regenerates the mesh via def.make(current)
and `obj.mesh.copyFrom(newMesh)` on the SAME object (undo stays the single
Add entry; document this Blender-matching behavior). The panel dismisses on:
any pointerdown in the viewport, any operator key (G/R/S/Tab/Shift+A...), or
Escape — simplest robust rule: a capture-phase listener that closes unless
the event targets the panel; panel owns and removes its listeners (AddMenu
close pattern). Only the LATEST add shows a panel.

## Acceptance criteria
1. Unit: every primitive's make() respects params (vert counts for segment
   changes, bounding radius for radius changes); circle with fill has 1 face
   and `vertices` verts; defaults unchanged vs current output (snapshot the
   current default cube/torus vert counts so existing scenes look identical).
2. e2e: Shift+A → Torus → panel appears; set majorSegments 12 via the panel →
   evaluated vert count matches 12×minorSegments; click in the viewport →
   panel gone; Ctrl+Z removes the torus entirely (single undo step).
3. All e2e suites (flock) + npm test + npx tsc --noEmit pass.

### Circle fill behavior (documented per spec)
`makeCircle(..., fillNgon=true)` (default) → one n-gon face + `vertices` verts.
`fillNgon=false` → the ring of verts with NO face — and, because EditableMesh
derives edges from faces, no edges either (a fill-less circle is invisible in
wireframe and has nothing for edit tools to grab). Fill stays ON by default so
fresh circles are usable; the bool is exposed but this limitation is expected.

### DEVIATION — Torus default params (deliberate, documented)
The spec's Primitive-params list gives Torus `minorRadius 0.4 / majorSegments 32
/ minorSegments 16`, but acceptance criterion 1 requires "defaults unchanged vs
current output … so existing scenes look identical", and the CURRENT
`makeTorus()` default is `minorRadius 0.25 / majorSegments 48 / minorSegments
12` (576 verts). These conflict. Every OTHER primitive's listed defaults already
match the current maker (Cube size 2, UV Sphere 32/16, Cylinder 32, Ico subdiv
2), so the torus numbers read as a mis-remembered default. I honored the
gradeable criterion + product invariant and set the Torus panel defaults to the
current maker values (0.25/48/12 → 576 verts). The panel still lets you dial in
0.4/32/16 (or the donut's values); only the *default* differs from the prose.
Named maker function signatures (makeCube/makeTorus/…) are unchanged — all their
existing callers/tests are untouched.

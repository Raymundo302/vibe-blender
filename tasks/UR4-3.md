# UR4-3 — Shift+A → Image submenu: image planes (Diffuse / Emit)

## Goal
New **Image ▸** category in the Shift+A add menu (after Mesh/Light, before Camera)
with two working items — **Diffuse…** and **Emit…** — each opening a file picker;
the chosen image becomes a textured plane in the scene. Diffuse = lit like any
material; Emit = shadeless, renders exactly as the image looks (blueprints/refs).

## Files
- EDIT `src/ui/addMenu.ts` — add the category (UR3-4 flyout machinery exists;
  follow the Mesh/Light pattern). A third item **HTML…** will be added by UR4-4;
  leave room but do NOT stub it.
- NEW `src/tools/imagePlane.ts` + tests — the logic, DOM-light (file input +
  Image decode isolated behind small functions so the geometry/material part is
  unit-testable).
- Material model: ADD an optional boolean `shadeless` to the scene material
  (default false) — grep the material type + DEFAULTS in
  `src/core/scene/objectData.ts` and the sceneJson serializer (v9 → v10 with the
  optional field; follow how v8→v9 added optional key fields; loading a v≤9 file
  must be byte-identically unaffected).
- Rendered-mode shader + F12 tracer: a `shadeless` material outputs its
  base/texture color DIRECTLY — no lights, no shadows (tracer: treat as emission
  of the texture color with no additional bounces gathered; Rendered: bypass the
  BRDF sum). Screen-space AO still multiplies in Rendered mode — acceptable,
  note it in the header.

## Behavior
- Click Diffuse…/Emit… → `<input type=file accept="image/*">` → read as dataURL,
  decode for natural width/height.
- Create a plane: XY footprint (Z-up world), height 2 world units, width
  2·(w/h), at the 3D cursor like every add. Name = file basename (sans ext).
  UVs: the plane primitive already ships unwrapped 0..1 — verify orientation
  matches the image right-side-up in Rendered mode (v-flip convention: check the
  app's existing image-texture convention in materialTab/bake — there is a known
  v-flip rule; follow it).
- Material (added to the scene material library, assigned to the plane):
  - name = basename; `texKind='image'`, `texDataUrl=<dataURL>`.
  - Diffuse: roughness 1, metallic 0, shadeless false.
  - Emit: shadeless true.
- ONE undo entry for object+material+assignment ("Add Image Plane") — single
  Ctrl+Z removes all of it (see joinCommand/separateCommand for multi-effect
  command precedent; if material-library add/remove has no command yet, build the
  minimal one inside your new command).
- Cancel the file picker → nothing happens, no undo entry.
- The Shift+A menu closes when the picker opens (the add commits on file chosen —
  keep the AddMenu teardown contract; the picker + commit logic lives in
  imagePlane.ts, invoked from the menu item).

## Out of scope
- No drag-and-drop import, no URL input, no image sequence.
- No changes to matcap/studio shading (textures show in Rendered/tracer only,
  as today).

## Acceptance criteria
- tsc clean; npm test green (serializer round-trip tests for `shadeless`,
  aspect/naming/material unit tests with a stubbed decode).
- e2e (new `e2e/image-plane.mjs`, unique E2E_PORT): drive `imagePlane`'s exported
  create function directly with a small data-URL PNG (bypass the picker — export
  a `createImagePlane(scene, undo, {dataUrl, name, w, h, mode})` used by both the
  picker path and tests): plane added with correct aspect + material wired
  (diffuse), emit variant has shadeless=true; single undo removes plane AND
  material; switch viewport to Rendered (`?shading=rendered` deep link exists) and
  pixel-check the emit plane shows the image's color unlit (use a solid-color
  PNG; compare against a diffuse plane in shadow — emit must be brighter/exact).
- Screenshot both variants in Rendered mode for the architect's eyes-on.
- If ambiguous or wrong, stop and report.

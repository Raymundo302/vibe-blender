# UR15-1 — Viewport raytraced mode (Rendered → Live | Raytraced, CPU/GPU engine)

## Goal
Blender-Cycles-style viewport rendering: the shading dropdown's **Rendered**
mode gains a sub-choice — **Live** (the current rasterized renderedPass,
unchanged) or **Raytraced** (the REAL path tracer progressively accumulating
in the viewport), with an Engine select **CPU | GPU** (GPU default when the
UR12 probe passes). Ray's words: "see the GPU rendering live."

## UI + prefs
- shadingMenu.ts: when the Rendered mode row is active, show beneath it (same
  indent family as section bodies): `Mode: [Live | Raytraced]` and, when
  Raytraced, `Engine: [GPU | CPU]` (GPU listed first + default; CPU fallback
  auto-selected with tooltip reason when the GPU probe fails).
- shadePrefs v6→v7: `renderedMode: 'live' | 'ray'` (default 'live'),
  `rayEngine: 'gpu' | 'cpu'` (default 'gpu'). Migration chain intact from v3.
- The Z-cycle and deep links treat Rendered as one mode (sub-choice persists).

## Viewport raytraced rendering (Renderer.ts + a new viewportRay module)
- When shadingMode==='rendered' && renderedMode==='ray': instead of the raster
  solid pass, draw the tracer's CURRENT accumulation as a fullscreen textured
  quad, then draw overlays ON TOP with a depth prime: rasterize the scene
  depth-only first (the wirePass prime shader exists) so gizmo/outline/cage/
  guides depth-test correctly over the traced image.
- **Accumulation loop** (the core): per rAF tick, if the accumulation is not
  converged (spp < cap, default 256), render a batch (GPU: start 1 spp/tick,
  adapt batch size to keep tick < ~24ms; CPU: one worker tile batch per tick —
  reuse the renderWindow progressive plumbing, slower is fine).
- **Reset triggers** (accumulate-while-static): camera view/proj change, scene
  content version (grep what the GPU repack keys on from UR12-3 and reuse
  EXACTLY that granularity), material/light/world edits, frameCurrent change,
  canvas resize, mode/engine switch. Missing a trigger = stale ghost image —
  the e2e attacks this.
- **Interaction degradation**: while the camera moves, drop to HALF resolution
  and keep resetting (1 spp — it looks noisy-but-live, Blender-like); on
  stillness, switch to full res and accumulate to the cap. LINEAR-upscale the
  half-res (the UR6 Framebuffer linear flag exists).
- A small viewport chip (same slot family as "Shading: …") shows
  "Raytraced · GPU · 37 spp". Sample count exposed on `__app` for e2e.
- Live mode must remain BIT-IDENTICAL to today (probe in e2e).

## Out of scope
Denoise, adaptive sampling, region render, overlays inside the ray image
(reflections of the gizmo etc. — obviously not), changing F12/Ctrl+F12.

## Acceptance criteria
- tsc clean; units green (prefs migration; batch-size adapter pure test).
- e2e (new e2e/viewport-ray.mjs, unique port; run on E2E_GPU=1 AND SwiftShader
  — CPU engine path can carry SwiftShader if the GL tracer is too slow there,
  document): (1) switch Rendered→Raytraced GPU: spp counter climbs, viewport
  converges to match an F12 GPU render of the same scene (mean-abs-diff of
  32² luminance < 0.05); (2) orbit the camera → spp RESETS and keeps rendering
  (no freeze, no stale ghost: probe a pixel that must change with the view);
  (3) edit a material color → reset + new color appears; (4) gizmo + selection
  outline visible OVER the traced image; (5) Engine CPU also converges (small
  res, few spp); (6) Live mode unchanged (pixel-compare a frame vs a
  pre-change baseline capture); (7) mode/engine persist across reload.
- Screenshots: raytraced viewport mid-convergence with gizmo visible, LOOKED
  at. Free-text report, FIRST LINE "STATUS: done"/"STATUS: blocked". NEVER end
  your turn to wait on a long run — poll in-command (the watchdog has killed
  two agents for this).

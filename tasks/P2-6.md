# P2-6 — Delete menu (X) + Merge at Center (M) in edit mode

Read `tasks/P2-CONVENTIONS.md` first.

## Goal
In edit mode, X opens a small menu at the pointer — Verts / Edges / Faces /
Merge at Center — applying the matching topology op to the selection with undo.
M applies Merge at Center directly (Blender muscle memory).

## Files
- NEW `src/ui/deleteMenu.ts` — copy the structure/style of `src/ui/addMenu.ts`
- `src/input/InputManager.ts` — wire X and M in `onEditModeKey`
- `e2e/edit.mjs` — append checks

## Design
- Menu entries call the EXISTING EditableMesh ops (`deleteVerts`, `deleteEdges`,
  `deleteFaces`, `mergeVertsAtCenter`) wrapped in `MeshEditCommand.capture`
  ('Delete Verts' / 'Delete Edges' / 'Delete Faces' / 'Merge at Center').
- What gets deleted comes from the CURRENT selection sets:
  - Verts entry: `sel.verts` expanded via `selectedVertIds(mesh)` when not in
    vert mode (delete the verts implied by the selection).
  - Edges entry: `sel.edges` (edge mode only; disable/omit the entry otherwise).
  - Faces entry: `sel.faces` (face mode only; disable/omit otherwise).
  - Merge: `selectedVertIds(mesh)` (needs ≥ 2).
- Empty selection: X does nothing (no menu, no crash).
- After the op: `sel.clearSelection()` (Blender leaves nothing selected after
  delete; after merge, select the surviving vert if in vert mode), `sel.prune(mesh)`.
- Menu closes on Escape / click-away, exactly like AddMenu.

## Out of scope
Dissolve, limited dissolve, edge-loop delete, merge-first/last variants.

## Acceptance criteria
1. e2e: edit mode, vert mode, select 2 corner verts via `__app`, M → 7 verts and
   the survivor selected; Ctrl+Z → 8. Face mode, select 1 face, X → menu →
   Faces → 5 faces, Ctrl+Z → 6.
2. X in edit mode never deletes the OBJECT (existing P2-1 e2e check keeps passing
   — the empty-selection case must early-return before the menu opens... note the
   existing check presses X with an empty element selection).
3. `npm run build`, `npm test`, `node e2e/edit.mjs` all pass.

## Result

Built `src/ui/deleteMenu.ts`: a self-contained `DeleteMenu` DOM widget mirroring
`AddMenu` (owns its element + listeners, closes on Escape / click-away, reuses the
shared `.add-menu` theme classes). It shows Verts / Edges / Faces / Merge at Center,
rendering entries that don't apply to the current element mode or selection as
disabled (greyed via inline opacity). Two exported helpers do the actual work so
both the menu and the M key share them: `deleteSelection(kind, …)` wraps the
existing `deleteVerts` / `deleteEdges` / `deleteFaces` ops in
`MeshEditCommand.capture` ('Delete Verts/Edges/Faces') and clears+prunes the
selection; `mergeAtCenter(…)` wraps `mergeVertsAtCenter` ('Merge at Center'),
needs ≥ 2 implied verts, and leaves the survivor selected in vert mode.

Wired into `InputManager.onEditModeKey`: X opens the menu at the pointer but
early-returns (no menu, never touches the object) when `selectedVertIds` is empty;
M runs `mergeAtCenter` directly. Appended P2-6 e2e checks to `e2e/edit.mjs`.

Verification: `npm run build` (tsc + vite) clean, `npm test` 93/93, `node e2e/edit.mjs`
all 51 checks pass — including AC1 (M: 2 verts → 7 + survivor selected, Ctrl+Z → 8;
X → menu → Faces: 6 → 5, Ctrl+Z → 6) and AC2 (X with empty selection opens no menu
and leaves the object intact; existing P2-1 in-edit-mode X check still green).

Deviations: none. Verts/Edges/Faces/Merge entries are shown-but-disabled when
inapplicable (spec allowed "disable/omit"). Delete-menu styling reuses the existing
`.add-menu` classes rather than adding CSS (theme.css was not in the allowed files).

# P3-4 — Shortcut cheat-sheet, splash, polish

Read `tasks/P3-CONVENTIONS.md` first. Runs LAST in the batch — it documents
every shortcut that exists at that point (read InputManager to enumerate them,
including P3 additions).

## Goal
F1 (or clicking a topbar "?" button) toggles a keyboard-shortcut overlay.
First visit shows a small splash card (dismiss on any click/key; remember via
localStorage `vibe-blender-splash-seen`). Status-bar idle hints.

## Files
- NEW `src/ui/helpOverlay.ts` — overlay with two-column shortcut table grouped
  by Object Mode / Edit Mode / Camera / File; data lives in an exported
  `SHORTCUTS` array (group, keys, description) so it stays greppable
- NEW `src/ui/splash.ts` — title, one-line pitch ("A Blender-style modeler,
  hand-built in TypeScript + WebGL2"), 3 bullet quick-start (orbit MMB, Tab edit
  mode, F1 shortcuts), auto-dismiss + localStorage
- `src/ui/topbar.ts` — "?" button (`data-action="help"`)
- `src/input/InputManager.ts` — F1 toggle (preventDefault; works in both modes;
  Escape closes overlay before anything else — check the activeOp branch)
- `src/ui/theme.css` — overlay/splash styles matching the Blender-dark theme
- `src/main.ts` — wiring
- `e2e/ship.mjs` — append checks

## Constraints
- The overlay must list EVERY shortcut wired in InputManager at the time you
  run — cross-check by grepping; a missing shortcut is a spec violation.
- Overlay and splash are DOM-only; no canvas changes. While the overlay is
  open, keyboard input must not leak to the viewport (F1 again or Escape closes).
- e2e runs with a fresh Chrome profile, so the splash WILL appear in e2e/smoke.mjs
  and e2e/edit.mjs runs — dismiss must be a no-op for their flows... which you
  cannot guarantee by hoping. Auto-dismiss the splash on the first canvas
  pointer/key event AND make e2e/ship.mjs assert both paths. If existing e2e
  files break because of the splash, add ONE line to their start (dismiss) —
  that's an allowed touch, disclose it in your Result.

## Acceptance criteria
1. e2e (ship.mjs): splash visible on fresh load; pressing a key dismisses it and
   sets localStorage; reload → no splash; F1 opens overlay listing (spot-check)
   "Ctrl+R", "Tab", "F1" entries; Escape closes it; G while overlay open does
   NOT start a move.
2. smoke.mjs and edit.mjs still pass end-to-end.
3. `npm run build`, `npm test`, all three e2e files pass.

## Result

Built the F1 shortcut cheat-sheet, first-visit splash, and topbar "?" button.

**What was built**
- `src/ui/helpOverlay.ts` — exported `SHORTCUTS` array (group/keys/description) and
  a `HelpOverlay` class. The array enumerates EVERY InputManager keymap branch,
  cross-checked against the handler: General (Tab, Z shading, Ctrl+Z, Ctrl+Shift+Z,
  F1, Enter/LMB confirm, Esc/RMB cancel), Object Mode (LMB, G, R, S, Shift+A,
  Shift+D, Alt+A, X), Edit Mode (LMB, 1/2/3, A, Alt+A, B, Ctrl+I, G, R, S, E, I,
  Ctrl+R, X, M), Camera (MMB/Shift+MMB/wheel), File (Ctrl+S, Ctrl+O). Overlay
  mounts lazily on `<body>`; backdrop-click and × close it.
- `src/ui/splash.ts` — `Splash` class + `SPLASH_SEEN_KEY`. Title, one-line pitch,
  3 quick-start bullets (MMB orbit / Tab edit / F1 shortcuts). Only renders when
  the localStorage flag is unset; `dismiss()` is idempotent and sets the flag.
- `src/input/InputManager.ts` — added a `help` constructor dep (structural type)
  and a guard at the very top of `onKeyDown`: while the overlay is open it
  swallows EVERY key (preventDefault) so nothing leaks to the viewport, and F1 /
  Escape close it — placed before the `activeOp` branch so Escape closes the
  overlay first. F1 toggles when closed.
- `src/ui/topbar.ts` — "?" button (`data-action="help"`) wired to `toggleHelp`.
- `src/main.ts` — creates `HelpOverlay`/`Splash`, passes help into InputManager,
  wires splash auto-dismiss on the first canvas pointerdown or any window keydown,
  and adds a small persistent `#idle-hint` ("F1 — shortcuts") beside the status
  readout.
- `src/ui/theme.css` — Blender-dark styles for `.help-overlay`/`.help-card`,
  `.splash`/`.splash-card`, and `#idle-hint`.
- `e2e/ship.mjs` — appended a P3-4 section: reloads to a fresh first-visit state
  and asserts splash visible + localStorage unset; a key dismisses it and sets the
  flag; reload → no splash; F1 opens the overlay listing Ctrl+R/Tab/F1; G while
  open does NOT move the cube; Escape closes; the "?" button toggles it too.

**Verification** — `npm run build` (tsc + vite) clean; `npm test` 126/126;
`node e2e/smoke.mjs`, `node e2e/edit.mjs`, `node e2e/ship.mjs` all green (run
sequentially). All acceptance criteria pass.

**Design note (no-op for existing e2e)** — the splash container is
`pointer-events: none`, so a click anywhere (even on the card) passes through to
the canvas, which reacts normally AND triggers the auto-dismiss. Because
smoke.mjs and edit.mjs interact immediately, the splash is gone before any state
is asserted, so neither file needed the "one dismiss line" the spec allowed —
they were left untouched and still pass.

**Deviations**
- Added a `General` group to `SHORTCUTS` beyond the spec's named
  Object Mode / Edit Mode / Camera / File groups. The cross-mode shortcuts (Tab,
  Z shading, undo/redo, F1, modal confirm/cancel) don't belong to any single
  mode; grouping them under General honors the "list EVERY shortcut" constraint
  more accurately than mis-filing them.
- "Status-bar idle hints" is implemented as a separate, non-interactive
  `#idle-hint` element pinned to the viewport's lower-right rather than writing
  into `#status`. Writing hints into `#status` would break smoke.mjs's strict
  `status === ''` assertions after a modal confirm, so the hint lives in its own
  element.

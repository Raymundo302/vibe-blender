# UR7-1 — HTML planes v2 core: object model, animated re-raster, keyable Play

## Goal
HTML planes become first-class objects: their page animates on the plane
(timed re-rasterization sampling the CSS animation clock), and **Play is a
keyable animation channel** so Ray can key: plane flies in (transform keys),
then the page's animation starts at frame N ("start the bouncy ball for 30
frames") — deterministic in viewport playback, scrubbing, AND Ctrl+F12 renders.

## Object model
New optional payload on SceneObject (objectData.ts, precedent: light/camera):
```ts
html?: {
  kind: 'file' | 'url';
  source: string;      // file: full HTML text (serialized into the scene); url: address
  pageW: number;       // raster viewport width CSS px (default 1024)
  pageH: number;       // raster viewport height CSS px (default 768)
  scrollY: number;     // page scroll offset in CSS px (consumed in UR7-2; serialize now)
  playing: boolean;    // the keyable play state
  fps: number;         // re-raster cap for live playback (default 8, range 1–15)
}
```
- `addHtmlPlaneFromText` (htmlPlane.ts) now stamps this payload (kind 'file',
  the source text) on the plane it creates. Old scenes load with `html`
  absent — unchanged planes. sceneJson bump per house convention (v13).
- Kind-aware duplicate must deep-copy the payload (grep the P8 duplicate).

## Page time model (the core design — implement exactly)
- `pageTime(frame) = Σ_{k = frameStart..frame-1} playing(k) / scene.fps`
  where `playing(k)` samples the object's `html.playing` F-curve (constant
  interp) at frame k — i.e. page-local seconds accumulate only while Play is
  keyed on. Implement as a pure exported function with unit tests; cache the
  prefix sums per (fcurve version) so scrubbing is O(1) amortized.
- Channel: `html.playing` via the channelPath system (sampler.ts `head.sub`
  split; light/material channels are the precedent — writeChannel maps 0/1 to
  the boolean). Keyed from the Properties UI (UR7-2's ● button; you wire the
  channel + sampler side and unit-test writeChannel round-trip).
- **Rasterize at a given time**: extend the rasterizer with
  `rasterizeHtmlAt(source, tSeconds, opts)` — inject a style forcing
  `animation-play-state: paused` and sampling pose via negative delay:
  `*{animation-play-state:paused !important; animation-delay:-<t>s !important}`.
  Document the v1 limitation: author-set animation-delays are overridden by
  the global sample clock (folded into one page clock).
- **Playback paths:**
  - Timeline playing or scrubbing (frameCurrent changes): re-raster (throttled
    to html.fps, latest-wins, skip if pageTime unchanged — e.g. playing=0
    span) and update the material texDataUrl in place via the existing texture
    invalidation path (UR4-4 Live reused it — same seam).
  - Free viewport (timeline stopped) with `playing=true` **and no
    playing-channel keys**: wall-clock advance at html.fps (preview behavior).
    With keys present, frameCurrent rules (no wall-clock drift).
  - Ctrl+F12 (animRender.ts): per rendered frame, AWAIT
    `rasterizeHtmlAt(source, pageTime(frame))` for every html plane before
    capturing — both engines. This is what makes the shot deterministic.
- Rasterization is async — a per-object in-flight guard (latest request wins,
  no queue buildup).

## Acceptance criteria
- tsc clean; npm test green. Units: pageTime integral (keys on/off spans,
  fractional fps), prefix-sum cache invalidation on curve edit, serializer
  round-trip v13 + old-file tolerance, rasterizeHtmlAt style injection string.
- e2e (extend e2e/html-plane.mjs): add a file plane whose HTML has a CSS
  animation moving a red block left→right across the full width; key
  html.playing on at frame 10 (off before); assert via rasterizeHtmlAt +
  pixel probes that frame 5 and frame 9 rasters are IDENTICAL (page clock
  frozen) and frame 40's block has moved; drive `__app` timeline scrub and
  assert the plane's material texDataUrl CHANGES between frames 10→40 and
  NOT between 1→9. Ctrl+F12 path: 3-frame PNG-seq render with the play key —
  frames differ post-key.
- Screenshot one animated pose for eyes-on. E2E_PORT as assigned. If the spec
  is ambiguous or wrong, stop and report.

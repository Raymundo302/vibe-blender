# UR5-5 — Real render resolution drives the camera frame + passepartout; camera spawn orientation

## Part A — Scene output resolution + a truthful camera frame
Today `src/ui/passepartout.ts` hardcodes `FRAME_ASPECT = 16/9`, disconnected from
anything the renderers do (they render at canvas size). Make the frame REAL:

1. **Scene render settings**: add `renderSettings: { width: number; height: number }`
   to the Scene (defaults 1920×1080), serialized in sceneJson (optional field on
   the current version — bump if the serializer conventions require; loading old
   files → defaults; follow the v8→v9 optional-field precedent).
2. **UI**: Resolution X / Y numeric fields — put them in the Camera tab
   (`src/ui/cameraTab.ts`) under a "Format" heading (closest thing we have to
   Blender's Output properties; note in your report if a better host exists).
   Undoable via the same command pattern other tab edits use.
3. **Consumers** (aspect = width/height everywhere):
   - `passepartout.ts`: frame rect uses the scene's aspect (keep `frameRect` pure,
     pass the aspect in).
   - **Look-through-camera projection**: when rendering through a camera
     (`cameraViewId` set), the camera's FOV must map to the OUTPUT frame, not the
     canvas: letterbox — the projection uses the render aspect, and the image is
     framed inside the passepartout rect exactly (what you see inside the frame =
     what renders). Find where the through-camera projection matrix is built in
     Renderer.ts and apply the aspect there. The passepartout then genuinely marks
     the render frame ("based on the resolution and the frame of the camera").
   - **F12 tracer + Ctrl+F12 animation**: render at renderSettings resolution
     (tracer output buffer + Render Window canvas + anim-render capture canvas),
     not the viewport canvas size. Check renderWindow.ts / animRender.ts for
     where dimensions come from today; keep a sane cap note if the Vega 7 chokes
     above 4K (just document, don't gate).
4. Viewport-only shading (matcap etc.) stays canvas-sized — this only concerns
   the camera frame + engine outputs.

## Part B — Camera spawns looking toward +Y
`Scene.addCamera` objects get identity rotation → in our Z-up world the camera
looks along −Z, straight down at the floor. New default: local −Z toward world
**+Y**, up staying +Z — i.e. rotation = +90° about X (verify sign against the
world-to-camera convention in the codebase; the acceptance test below is the
authority). Apply it in the Shift+A add path so the camera object's transform
carries the rotation (NOT inside addCamera's data — grep for other addCamera
callers: scene loads must not get double-rotated; the spawn rotation belongs
where the add menu creates it, next to the cursor-position logic in
addMenu.commitAdd — set rotation only for kind==='camera'). Numpad0 through a
fresh camera must show the horizon, not the floor.

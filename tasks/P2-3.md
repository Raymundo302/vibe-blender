# P2-3 — G/R/S on selected elements

Read `tasks/P2-CONVENTIONS.md` first.

## Goal
In edit mode, G/R/S run modal move/rotate/scale on the selected elements
(via `editMode.selectedVertIds(mesh)`), with the same feel as the object-mode
operators: X/Y/Z axis locks, numeric input, LMB/Enter confirm, RMB/Esc cancel,
live status text.

## Files
- NEW `src/tools/editTransform.ts` — three operators (or one parameterized class)
- `src/input/InputManager.ts` — wire G/R/S inside `onEditModeKey`
- `e2e/edit.mjs` — append checks

## Design (follow exactly)
- Model on `src/tools/translate.ts` / `rotate.ts` / `scale.ts` — same Operator
  interface, same status-bar format, reuse `src/tools/numericInput.ts`.
- Pivot = centroid (median) of the affected verts, in WORLD space for the
  pointer-plane math; convert final deltas to LOCAL space per the conventions doc
  before `setVertCo`.
- Rotate: about the view-forward axis through the pivot by default; X/Y/Z lock
  rotates about that world axis. Scale: uniform from pointer-distance ratio to the
  pivot's screen position; X/Y/Z lock scales along one world axis.
- Undo per the "modal GEOMETRY tools" pattern in the conventions doc — command
  name 'Move' / 'Rotate' / 'Scale'.
- If the selection is empty, `start()` returns false (no dead modal state).
- After confirm/cancel call `sel.touch()` so the overlay refreshes.

## Out of scope
Proportional editing, snapping, per-element pivots, gizmos in edit mode.

## Acceptance criteria
1. Unit tests for the pure math you add (e.g. local-space delta conversion,
   centroid). Operator flow tests follow the pattern in `src/tools/operator.test.ts`.
2. e2e: edit mode, select-all (A), G, move pointer, click-confirm → a known vert's
   position changed; Ctrl+Z → restored. S with numeric input `2` + Enter → vert
   coordinates doubled from the pivot; Ctrl+Z restores.
3. Object-mode G/R/S behavior unchanged (existing smoke.mjs still passes:
   `node e2e/smoke.mjs`).
4. `npm run build`, `npm test`, `node e2e/edit.mjs` all pass.

## Result
Built `src/tools/editTransform.ts` — `EditTranslateOperator` (G), `EditRotateOperator` (R),
`EditScaleOperator` (S) sharing a base class, modeled on `translate.ts`/`rotate.ts`/`scale.ts`
(same Operator interface, status format, axis locks, `NumericInput`). Pivot = world-space
centroid of `editMode.selectedVertIds(mesh)`; rotate/scale offset math is in world space and
positions are written back via `invMatrix.transformPoint`; translate converts the world delta
with `invMatrix.transformDir`. Undo follows the modal-GEOMETRY pattern (capture before, restore,
push `MeshEditCommand.capture`), command names 'Move'/'Rotate'/'Scale'. Empty selection → `start()`
returns false; confirm/cancel call `sel.touch()`. Wired G/R/S into `InputManager.onEditModeKey`.
Exposed pure helpers `centroid` and `worldDeltaToLocal`.

Verified: `npm run build` (clean tsc + bundle), `npm test` (85 tests incl. new
`src/tools/editTransform.test.ts` — centroid, worldDeltaToLocal, scale-2 doubling, undo/cancel,
empty-selection guard), `node e2e/edit.mjs` (24/24 incl. new G-move+undo and S"2"+Enter+undo
checks), `node e2e/smoke.mjs` (12/12 — object-mode G/R/S unchanged).

Deviation: added a new sibling test file `src/tools/editTransform.test.ts` (not in the Files list)
because acceptance criterion 1 requires unit tests and the reference pattern lives in a sibling
test file. No other out-of-scope files touched.

# P3-3 ‚Äî Shading modes: matcap / wireframe / studio

Read `tasks/P3-CONVENTIONS.md` first. Runs after P3-1.

## Goal
Z cycles the viewport shading: **matcap** (current) ‚Üí **wireframe** (edges only,
no solid fill) ‚Üí **studio** (flat lambert with a fixed key light) ‚Üí matcap.
A topbar chip shows the current mode.

## Files
- NEW `src/render/passes/studioPass.ts` ‚Äî lambert shader pass, same shape as
  `meshPass.ts` (begin/setObject, normal matrix, `layout(location = 0/1)`)
- NEW `src/render/passes/wirePass.ts` ‚Äî draws every visible object's unique
  edges (`meshToRenderData(...).edgePositions`) as dark lines; cache per
  (object id, mesh.version) like `Renderer.gpuMesh` does ‚Äî you will need a
  second cached VertexArray; add it to the renderer's cache entry type
- `src/render/Renderer.ts` ‚Äî `shadingMode: 'matcap' | 'wireframe' | 'studio'`
  (default 'matcap'); branch the solid pass on it. In wireframe mode the
  outline/edit passes still work (mask uses triangles as today).
- `src/input/InputManager.ts` ‚Äî Z key (both object and edit mode, no modifiers)
- `src/ui/topbar.ts` ‚Äî mode chip (`data-action="shading-mode"`, clicking it also
  cycles)
- `e2e/ship.mjs` ‚Äî append checks

## Studio shader
Two hardcoded view-space directional lights (key `normalize(0.4, 0.6, 0.7)`
intensity 0.9 warm-white, fill `normalize(-0.5, -0.2, -0.4)` intensity 0.35
cool) + 0.12 ambient; base color `vec3(0.62)`; no specular. Half-lambert is NOT
wanted ‚Äî plain `max(dot(n,l), 0)`.

## Acceptance criteria
1. e2e: `__app.renderer.shadingMode` cycles matcap‚Üíwireframe‚Üístudio‚Üímatcap on
   Z presses; topbar chip text follows; clicking the chip cycles too; screenshot
   each mode to /tmp (matcap vs studio vs wireframe must differ ‚Äî compare PNG
   byte lengths differ as a cheap proxy and save all three for human review).
2. Edit mode still fully works in wireframe shading (Tab + A + G smoke check
   in e2e).
3. `npm run build`, `npm test`, all three e2e files pass.

## Result

Built viewport shading-mode cycling: **matcap ‚Üí wireframe ‚Üí studio ‚Üí matcap**,
driven by the `Z` key (both object and edit mode, no modifiers) and by a
clickable topbar chip.

**What was built**
- NEW `src/render/passes/studioPass.ts` ‚Äî flat two-light lambert pass, same
  shape as `meshPass` (`begin`/`setObject`, view-space normal matrix,
  `layout(location = 0/1)`). Two hardcoded view-space directional lights (key
  `normalize(0.4,0.6,0.7)` √ó0.9 warm-white, fill `normalize(-0.5,-0.2,-0.4)`
  √ó0.35 cool) + 0.12 ambient, base `vec3(0.62)`, plain `max(dot(n,l),0)`, no
  specular.
- NEW `src/render/passes/wirePass.ts` ‚Äî position-only shader drawing dark edge
  lines (no fill). The per-object edge `VertexArray` is cached in the renderer.
- `src/render/Renderer.ts` ‚Äî added `ShadingMode` type + `shadingMode` field
  (default `'matcap'`) + `cycleShadingMode()`. The `GpuMesh` cache entry gained
  an `edges: VertexArray` (built from `meshToRenderData(...).edgePositions`,
  disposed alongside `triangles`, keyed by the same `mesh.version`). The solid
  pass now branches on `shadingMode`. Outline/edit-cage/gizmo overlays are
  untouched (mask still uses triangles), so edit mode works in every shading mode.
- `src/input/InputManager.ts` ‚Äî `Z` keybind (before the edit-mode branch so it
  works in both modes; plain Z only, Ctrl+Z stays undo).
- `src/ui/topbar.ts` ‚Äî shading chip (`data-action="shading-mode"`) that shows the
  capitalized mode and cycles on click.
- `e2e/ship.mjs` ‚Äî appended P3-3 checks (cycle order, chip text follows, click
  cycles, three /tmp screenshots whose byte lengths differ, and a wireframe-mode
  edit smoke: Tab + A (8 verts) + G-move confirm).

**Verification**
- `npm run build` ‚Äî clean (tsc + vite).
- `npm test` ‚Äî 126/126.
- `node e2e/smoke.mjs`, `node e2e/edit.mjs`, `node e2e/ship.mjs` ‚Äî all green
  (ship: 33 checks). Screenshots saved: `/tmp/p3-3-matcap.png` (130690 B),
  `/tmp/p3-3-wireframe.png` (136358 B), `/tmp/p3-3-studio.png` (130125 B) ‚Äî all
  three differ, confirming the modes render distinctly.

**Deviation**
- `src/main.ts` was edited (one line: `new Topbar(scene, renderer, {...})`).
  It is not in the spec's Files list, but the shading chip must read/mutate
  `renderer.shadingMode`, and P3-CONVENTIONS explicitly prescribes passing new
  topbar deps through the constructor ("no globals"). The alternative (reaching
  through `window.__app`) would violate that convention. The main.ts change is
  the minimal integration wiring required by the spec's own topbar requirement.

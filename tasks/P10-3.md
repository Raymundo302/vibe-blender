## Tokenization
Map every color in the css files onto the 11 tokens (bg, panel, panel2, input,
border, text, text-dim, accent, accent-contrast, hover, chip) with the CLAUDE
theme's values as the var() fallbacks — so the app looks IDENTICAL to today
with no theme applied (that's the regression bar: screenshot-compare or spot
pixel checks in your e2e). Rgba shadows/overlay dims may stay literal. If a
color genuinely fits no token (e.g. the snap magnet green), leave it literal
and list it in Result.

## Picker
Topbar button → popup (AddMenu widget pattern): one row per themes() entry
showing name + tagline + a small swatch strip (panel/accent/text), current one
highlighted; click applies + persists (applyTheme handles both) and closes.
Theme choice is NOT undoable (app preference, not scene state) — document.

## Acceptance criteria
1. Claude theme (default boot) is pixel-identical to today in the key spots
   your e2e checks: topbar background, panel background, accent button color
   (getComputedStyle), viewport clear pixel.
2. e2e: open picker, switch to ice-blue → body/panel computed colors change to
   light values AND a viewport corner pixel reads light grey (readPixels after
   a frame); switch to retro-dos → dark green; reload → theme persisted;
   switch back to claude → original values.
3. Every theme: text vs panel contrast ratio ≥ 4.0 (compute in a unit test
   over themes(); WCAG-ish luminance math in the test file).
4. All e2e suites pass under flock (donut.mjs excluded — architect's final
   sweep) + npm test + npx tsc --noEmit.

# UR7-3 — URL web planes: add dialog (Load / Open), live iframe portal, pause

Builds on UR7-1/2 (in your tree). The "website portal" feature.

## A. Add dialog replaces the two Shift+A HTML items
Shift+A → Image ▸ **HTML / Website…** (ONE item now) opens a small centered
dialog (house theme):
- A text input for a web address (placeholder "https://…"), button **Load** →
  creates a URL plane (part B).
- Button **Open…** → the existing local-file picker path (UR4-4 → UR7-1 file
  plane; keep the on-disk live-reload watcher when showOpenFilePicker exists).
- Esc/click-outside cancels. Menu teardown contract as always.

## B. URL plane = live portal (document the platform truth loudly)
A cross-origin website cannot be drawn into WebGL (canvas tainting). The live
option is a PORTAL: a real `<iframe>` overlaid on the viewport, transform-
matched to the plane each frame.
- Object: html payload kind 'url', source = address, `playing=true` by default
  ("a website will be live unless I pause it").
- Portal element: an iframe inside a `#viewport-wrap` overlay layer with
  `transform-style: preserve-3d`; per frame (frame-loop update, like
  passepartout) compute the CSS `matrix3d` mapping the plane's world quad
  through the current view+projection to screen (classic CSS3D sync — derive
  perspective from the projection matrix; the iframe's CSS size is pageW×pageH
  and the matrix scales it onto the plane). Hidden when the plane is behind
  the camera or its object hidden. **Known limits — document in module header
  AND a one-line hint in the Properties section: no occlusion (portal draws
  over the 3D scene), invisible in F12/Ctrl+F12 renders and viewport
  screenshots (GL never sees it).**
- Interaction: the overlay layer is pointer-events:none EXCEPT in page mode
  (UR7-2 Tab) — in page mode the iframe gets pointer events, so hover/scroll/
  click work FOR REAL inside the site. Tab out → viewport input back.
- **Pause** (the UR7-2 ▶/⏸ toggle, same channel): portal hides. In its place
  the plane's texture shows: (a) if the URL is fetchable with CORS (try
  `fetch(url)` → text → the UR7-1 raster pipeline), the rasterized page;
  (b) otherwise a neutral "Paused web portal — <host>" card (errorCard-style,
  info-grey not red). Play again → portal returns. Keying playing works like
  any channel (portal follows the sampled value during playback — a portal
  blinking on at frame N is acceptable v1).
- Serialization: url + payload round-trip; on LOAD a url plane starts paused
  with its card/raster (no surprise network/portal on file open — document;
  user presses ▶).
- Security niceties: iframe `sandbox="allow-scripts allow-same-origin
  allow-forms"` (no top-navigation, no popups); referrerpolicy no-referrer.
  Sites that refuse framing (X-Frame-Options) will show an empty portal —
  detect nothing; just document it and let the pause-raster path be the
  fallback the user discovers (hint text in Properties mentions it).

## Acceptance criteria
- tsc clean; npm test green (matrix3d math unit-tested against a known
  camera/plane pose — pure function screenMatrixForPlane(world, view, proj,
  viewportPx) with exported expected values; dialog construction pure parts).
- e2e (new e2e/web-portal.mjs, unique port): (1) Shift+A → Image → the dialog
  opens; typing a `data:text/html,<h1>hi</h1>`-style or same-origin URL (serve
  a test page from the vite dev server — /research/bouncy-ball.html works) and
  Load → url plane + portal iframe exists with a matrix3d transform; orbit the
  camera → the matrix CHANGES (probe style.transform). (2) Tab (page mode) →
  iframe has pointer-events; Tab out → none. (3) Pause → portal hidden, same-
  origin URL rasters onto the plane (pixel probe); Play → portal back.
  (4) Scene save→load → plane present, paused, no iframe until ▶. (5) The
  UR7-1/2 file-plane checks still green.
- Screenshot: portal showing bouncy-ball.html on the plane at an angle.
- E2E_PORT as assigned. Stop and report if ambiguous — especially if the
  matrix3d sync fights the workspace layout; report rather than hack.

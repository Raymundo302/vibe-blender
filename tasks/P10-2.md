## Lock to View
Today, orbit/pan/zoom while in camera view EXITS the view (P8-2). New rule:
if the viewed camera's `lockToView` is true, navigation stays in camera view
and MOVES THE CAMERA instead:
- On entering camera view (Numpad0) with lock on — or when the checkbox turns
  on while in camera view — initialize a private orbit rig from the camera
  pose: eye = camera position, forward from objectForward, target = eye +
  forward × d where d = the camera's distance to the world origin projected
  onto forward, clamped to 1..50 (document; Blender uses the view's own dolly
  target, this is our stand-in).
- Orbit/pan/zoom mutate the rig (reuse OrbitCamera math — you may instantiate
  an OrbitCamera as the rig; do NOT touch the main viewport OrbitCamera).
- Each nav event writes the rig pose back to the camera object's transform
  (position = rig eye, rotation = look-at quaternion; reuse/extract the
  cameraToView helper math from P9-6).
- Undo: ONE TransformCommand per camera-view session, pushed when leaving
  camera view (or toggling lock off) if the pose changed — continuous nav must
  not spam the undo stack (document this Blender-ish behavior).
- lockToView false → today's exit-on-orbit behavior, unchanged.

## Passepartout
When (and only when) the renderer is in camera view: a DOM overlay inside
#viewport-wrap that dims everything outside the render frame — the largest
16:9 rect that fits centered in the viewport (16:9 = the tracer's output
aspect). Four absolutely-positioned divs (top/bottom/left/right) with
`background: rgba(0,0,0,0.55)` + a 1px border rect on the frame (use theme
var --vb-border or a dedicated --vb-passepartout if you add it to BOTH themes
in themes.ts — smaller diff wins). Recompute on resize + camera-view
enter/exit (a per-frame update() call from main.ts's frame loop is fine and
matches the panels' pattern — you may add ONE wiring line to main.ts for it).
Pointer-events: none — it must never eat viewport input.

## Acceptance criteria
1. Unit: rig round-trip (camera pose → rig → pose is identity within epsilon);
   orbiting the rig keeps the target fixed; pose-changed detection.
2. e2e: camera with lockToView OFF: MMB drag exits camera view (existing
   behavior asserted). Toggle lock ON via the Camera tab: Numpad0, MMB drag →
   STILL in camera view (cameraViewId set) and the camera object's transform
   changed; leaving camera view pushes exactly ONE undo entry; Ctrl+Z restores
   the original pose. Passepartout: divs present + cover the expected pixels
   while in camera view (check a corner pixel is overlaid and a center pixel
   is not), absent outside camera view, and clicks still select through the
   frame area.
3. All e2e suites pass under flock (donut.mjs excluded — architect's final
   sweep) + npm test + npx tsc --noEmit.

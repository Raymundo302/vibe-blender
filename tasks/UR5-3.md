# UR5-3 — Render Animation: path-traced engine + MP4/format menu

## Goal
Ctrl+F12 (`src/renderEngine/animRender.ts`) currently records the REALTIME
viewport (Rendered mode) to WebM or a PNG-sequence zip. Ray wants to make final
videos with the actual render engine: add an **Engine** choice — Viewport
(current) / **Path Traced** — and widen formats to WebM / **MP4 (when the browser
supports it)** / PNG sequence.

## Engine: Path Traced
- Modal gains `Engine: [Viewport | Path Traced]`; choosing Path Traced reveals a
  `Samples` numeric field (default 64, range 8–1024) — samples per pixel per
  frame before the frame is captured.
- Per frame: pose with `applyAnimation` (as today), then run the EXISTING tracer
  (`src/renderEngine/tracer.ts` + `worker.ts` — the F12 path) to N spp for the
  active camera at the current output resolution, WITHOUT opening the Render
  Window UI: find the seam renderWindow.ts uses to drive the worker and reuse it
  headless (extract a small driver if needed — do not fork the tracer).
  The traced RGBA buffer feeds the same two sinks:
  - PNG sequence: pixels → offscreen canvas → toBlob (existing path).
  - WebM/MP4: draw the traced frame onto the offscreen canvas the
    MediaRecorder's captureStream records (the stream must come from whatever
    canvas holds the final frame — restructure the capture source so BOTH
    engines feed one recording canvas; viewport engine draws/copies the viewport
    canvas into it, keeping current behavior pixel-identical).
- Progress UI: `frame i/total · sample s/N` and Cancel must abort mid-frame
  (terminate/ignore the in-flight tracer batch, restore state — follow the
  existing cancel/restore contract exactly).
- Determinism: the tracer is seeded — per-frame seeds should differ (seed by
  frame index) so static shots don't repeat identical noise (check how F12 seeds
  and do the least surprising thing; document).

## Formats
- Format select: WebM (default), MP4, PNG sequence. MP4 appears ONLY when
  `MediaRecorder.isTypeSupported('video/mp4;codecs=avc1.42E01E')` (or a
  functioning variant — probe a shortlist at modal-open, pick the first
  supported; hide the option when none). Download extension/mime accordingly.
- PNG sequence stays the "exact pixels" path; add a one-line hint in the modal
  under the format row: "PNG → assemble with: ffmpeg -framerate {fps} -i
  frame_%04d.png out.mp4".

## Files
`src/renderEngine/animRender.ts` (+ test file), `renderWindow.ts` only for
extracting the headless driver seam, modal CSS if needed. Nothing else.

## Acceptance criteria
- tsc clean; npm test green (pure parts: format-probe shortlist logic with a
  stubbed isTypeSupported, per-frame seed derivation, frameCount untouched).
- e2e (extend the suite that covers Ctrl+F12 today — find it; else new
  `e2e/anim-render.mjs`): (1) path-traced PNG-seq render of a 2-frame range at
  tiny spp (8) and small canvas → zip blob produced, both frames non-black and
  DIFFERENT when an object is keyed to move; (2) cancel mid-run restores
  frameCurrent + shading mode; (3) WebM viewport path still produces a non-empty
  blob (regression); (4) MP4 option hidden/shown per the probe (assert menu state
  matches `MediaRecorder.isTypeSupported` in that browser).
- Headless Chrome may lack MP4 — the probe test must assert CONSISTENCY, not a
  specific availability.
- Report the real-GPU spot check (`E2E_GPU=1`) result for the path-traced run.
- If ambiguous or wrong, stop and report.

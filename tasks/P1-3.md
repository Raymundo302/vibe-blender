# P1-3 — Translate gizmo (move arrows, picking-based handles)

Read `tasks/P1-CONVENTIONS.md` first. This is the hardest task in the batch —
read `src/render/Renderer.ts`, `src/render/passes/pickingPass.ts`,
`src/render/passes/gridPass.ts` (for a simple line-drawing pass to crib from),
`src/input/InputManager.ts`, and `src/tools/translate.ts` before writing code.

## Files
- `src/render/passes/gizmoPass.ts` — new
- `src/render/Renderer.ts` — integrate gizmo draw + extend picking
- `src/input/InputManager.ts` — gizmo drag handling in pointer handlers
- `src/tools/translate.ts` — minimal: optional constructor arg to preset the axis lock
- `src/tools/gizmo.test.ts` — new, only if you have logic worth unit-testing (screen-scale math); otherwise skip

## Rendering
- When the scene has an active object AND no modal operator is running, draw a
  translate gizmo at the active object's position, on top of everything
  (clear depth before the gizmo draw, after outlines).
- Three axis arrows: X red (#e35a5a-ish), Y green, Z blue — line shaft + small
  cone or pyramid tip is enough (build the geometry procedurally in gizmoPass).
- Constant screen size: scale the gizmo by
  `distance(camera.eye, origin) * tan(fovY / 2) * K` with K ≈ 0.18.
- InputManager must tell the Renderer whether an operator is modal — add a tiny
  hook (e.g. `renderer.gizmoVisible = …` set from InputManager, or pass a flag
  into `render()`; pick the least invasive and document it).

## Picking
- Reserve gizmo pick ids far above object ids: `const GIZMO_PICK_BASE = 0xF00000`
  (X = base, Y = base+1, Z = base+2). Export from gizmoPass.
- Change `Renderer.pick` to return a discriminated union:
  `type PickResult = { kind: 'object'; id: number } | { kind: 'gizmo'; axis: 'x' | 'y' | 'z' }`
  (null for background). Update the existing call site in InputManager
  accordingly — selection behavior for objects must not change.
- During the pick render, draw the gizmo handles LAST with depth cleared
  (they win over objects), each shaft+tip in its axis' id color. Give the pick
  geometry generous thickness (a box around each shaft ~6px worth) so thin
  lines are grabbable.

## Drag behavior
- LMB down on a gizmo handle: do NOT change selection. Instead start
  `new TranslateOperator(axis)` (add an optional `constructor(axis?: 'x'|'y'|'z')`
  that presets the existing axis-lock field — no other translate changes) via
  `startOperator`, then route pointermove to it as usual.
- Release LMB → confirm (push undo). Esc/RMB before release → cancel.
  This differs from keyboard-G (which confirms on click); track a
  `gizmoDrag` flag in InputManager for the pointerup-confirms path.

## Acceptance criteria (headless CDP)
1. With the cube selected, pixels near its origin show the three arrows (screenshot sanity via `Page.captureScreenshot` — assert the gizmo pass ran by probing `__app` state or reading pick results at arrow-tip coordinates).
2. `Renderer.pick` at a point on the X arrow returns the gizmo/x result (expose enough on `window.__app` to assert this — renderer is already there).
3. Drag on the X arrow moves the cube along world X only (y and z stay 0 within 1e-6); position changes are confirmed on release and Ctrl+Z restores.
4. Clicking an object (not a handle) still selects it; clicking background still deselects; keyboard G flow unchanged (`node e2e/smoke.mjs` must still pass all 12 checks).
5. Gizmo hidden while a G/R/S operator is modal and when nothing is selected.
6. `npm run build` and `npm test` pass.

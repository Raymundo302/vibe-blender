## Outliner
Group rows under collection headers (Blender-style): a header per collection
(▸/▾ expand triangle persisted per session in the panel instance, 📁-ish glyph,
name, eye toggle, ✕ delete) with member object rows indented beneath; root
objects listed after (or before — match Blender: collections first). "New
Collection" button at the top. Double-click header renames (existing rename
pattern). Eye toggles collection.visible; ✕ removes the collection (members
drop to root — core behavior). Selecting/renaming/hiding OBJECT rows must keep
working exactly as today (frozen e2e asserts them).

## Move to Collection (M, object mode)
M with a selection (InputManager object-mode branch — M is currently edit-mode
merge only; keep that intact) opens a popup listing collections + "New
Collection" + "Scene Root" (null). Choosing assigns EVERY selected object's
collectionId. AddMenu is the widget pattern to copy.

## Undo
collectionCommands.ts: CreateCollectionCommand, DeleteCollectionCommand
(restores membership of every ex-member), MoveToCollectionCommand (before/after
map), SetCollectionVisibilityCommand. Every UI mutation goes through one.

## Acceptance criteria
1. Unit: each command round-trips under undo/redo (delete restores members;
   move restores per-object previous assignment).
2. e2e: create collection via outliner button; M moves 2 selected objects in;
   outliner shows them nested; eye off → objects vanish from the viewport
   (probe via renderer pick at a known-projected point or evaluate
   scene.effectiveVisible); Ctrl+Z chain restores everything; delete
   collection returns rows to root.
3. All e2e suites pass under flock (donut.mjs EXCLUDED — the architect runs it
   in the final sweep) + npm test + npx tsc --noEmit.

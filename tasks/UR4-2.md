# UR4-2 — Edge slide: past-the-end + proximity rail pick; G cycle → Normal move

Builds ON TOP of UR4-1 (already merged in your tree: virtual modal pointer,
continuousGrab). Three parts.

## A. Edge slide slides past the edge, proximity-picked
`src/tools/edgeSlide.ts` today maps horizontal pixels → t clamped to [-1,1].
Replace the mapping:
- **Control vert**: the selected vert whose projected screen position is nearest
  the pointer at `start()` (project with the same camera/world→screen path other
  tools use — find the existing projector helper; knife/gizmo have one).
- Each of the control vert's rails (a/b) defines a screen-space LINE (project
  `base` and `base + dir·length` through the edit object's world matrix).
- On every move: project the (virtual) pointer onto each rail line; pick the
  NEAREST line (perpendicular screen distance) — "proximity based, the closest
  line the mouse is at snaps to it"; t = signed parameter along that rail
  (projection scalar / rail screen length), mapped so t>0 = rail a, t<0 = rail b.
- **UNCLAMP t**: |t| may exceed 1 — verts continue along the rail's extension
  ("slide past the edge into the extended edge"). Clamp only to ±100 (sanity).
  `slidePosition` already extrapolates linearly — verify and keep.
- All selected verts still share the single t (existing behavior).
- Guides (UR3-3) stay; extend the drawn guide length to ±2L so the extension the
  vert can now reach is visible.
- Keep numeric input override; status shows t and which rail is active is NOT
  needed.
- Degenerate cases: control vert with one rail → only that line; zero rails →
  t stays 0 (existing). Rails parallel in screen space (edge-on view) → nearest
  by distance still well-defined; if a rail projects to a point (length < 2px),
  exclude it from picking.

## B. Normal move operator
NEW `src/tools/normalMove.ts` + tests — `NormalMoveOperator` ("Normal Move"):
- At `start()`: per selected vert, capture its vertex NORMAL (area-weighted
  average of adjacent face normals, normalized; grep for an existing vertex-
  normal helper — meshToGpu/recalcNormals have this logic; REUSE, don't fork) and
  its start position. **Normals are captured ONCE and locked for the whole modal**
  — moving verts changes face normals, but the captured vectors must NOT be
  recomputed mid-drag (Ray: "keep the vector locked till the move is done").
- A single scalar distance d, driven by pointer motion along a screen direction:
  use the same horizontal-pixel mapping edge slide used (pixels / (0.25·viewport
  width) × a world scale of 1) — simple and predictable; numeric input overrides.
- Every selected vert: `pos = start + normal·d` (LOCAL space — face normals from
  local coords are local; no world transform needed for the offset itself).
- Undo/confirm/cancel: exactly the EdgeSlideOperator pattern (capture, restore,
  MeshEditCommand). continuousGrab = true. Boundary/wire verts with no faces:
  normal = zero → vert stays (fine).

## C. G cycles Move → Edge Slide → Normal Move → Move
`src/input/InputManager.ts`: today a second G during edit-mode Move swaps to
EdgeSlide via the `slideRequested` sentinel (~line 687). Generalize:
- Replace the one-off sentinel with a `cycleRequested` sentinel available on the
  edit Move, EdgeSlide, and NormalMove operators (same mechanism: onKey('g')
  sets it, returns true).
- InputManager: on cycle request, cancel current (restore, push nothing) and
  start the next in [EditMove → EdgeSlide → NormalMove → EditMove …], handing
  over the SAME selection. Status line already shows each op's name.
- Object-mode G is untouched (no cycle).

## Out of scope

- No per-vert individual origins beyond what's specified.
- Don't touch UR4-1's modalPointer internals.

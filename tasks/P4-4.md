# P4-4 — Modifier tab UI + scene format v2

Read `tasks/P4-CONVENTIONS.md` first. Runs after P4-3 (uses registerPropertiesTab).

## Goal
A Modifiers tab (wrench '🔧') in the properties editor: add/remove/reorder/
enable/apply modifiers on the active object with a generic param UI, all
undoable. Scene files gain modifiers (format version 2).

## Files
- NEW `src/ui/modifierTab.ts` — the tab (registered via registerPropertiesTab)
- `src/io/sceneJson.ts` + `src/io/sceneJson.test.ts` — format v2
- `src/main.ts` — import the tab module (side-effect registration)
- `e2e/ship.mjs` — append checks

## Design
- "Add Modifier" <select> listing `modifierTypes()` (placeholder first option);
  choosing one pushes `ModifierStackCommand.capture('Add Modifier', obj, () =>
  obj.modifiers.push(createModifier(type)))`.
- Each stack entry (top-to-bottom = evaluation order): name (editable input),
  enable checkbox, ▲/▼ reorder buttons, ✕ remove, and **Apply** (enabled only
  on the FIRST entry; constructs `new ApplyModifierCommand(obj, mod)` and
  pushes it). Every mutation goes through ModifierStackCommand.capture (except
  Apply) — never touch obj.modifiers without one.
- Params render generically from `mod.fields()`: number/int → <input
  type=number> (step/min/max from the field), bool → checkbox, axis → X/Y/Z
  select. Param edits are undoable — capture ONE command per commit (input
  'change' event, not every keystroke), named 'Edit Modifier'.
- Empty state: "No modifiers" hint. No object selected: "No active object".
- Refresh via the panel signature-diff pattern (rebuild the list only when
  `activeObject?.id`/`modifiersVersion` changed; never rebuild DOM holding focus).

## Scene format v2
- `"version": 2`; each object gains `"modifiers": [{ "type", "name",
  "enabled", "params" }]`. Loader accepts v1 (modifiers = []) AND v2; unknown
  modifier types in a file throw the standard readable error before any
  mutation. Serialization order stays deterministic.

## Acceptance criteria
1. Unit: sceneJson round-trips a stack (2 modifiers, custom params/enabled) —
   NOTE: register a tiny test-local modifier via registerModifier in the test
   file; do not depend on P4-5's real modifiers existing yet.
2. e2e: with the cube active, add a modifier from the dropdown (whatever types
   exist by then — use `window.__app` to check; if none are registered yet,
   register a test one through the console path is NOT possible — instead gate:
   if `modifierTypes()` is empty, assert the empty dropdown renders and skip
   the stack checks with a log line).
3. Undo probes: add → Ctrl+Z removes; param edit → Ctrl+Z restores value.
4. All four e2e suites + npm run build + npm test pass.

## Result

Built (2026-07-06):
- NEW `src/ui/modifierTab.ts` — Modifiers properties tab (icon 🔧), registered via
  `registerPropertiesTab` (imported for side-effect in `src/main.ts`). Add-Modifier
  dropdown from `modifierTypes()` (disabled placeholder first). Each stack entry shows
  enable checkbox, editable name, ▲/▼ reorder (disabled at ends), ✕ remove, and Apply
  (enabled only on the first entry). Every stack mutation (add/rename/toggle/reorder/
  remove) goes through `ModifierStackCommand.capture`; Apply pushes `ApplyModifierCommand`.
  Params render generically from `mod.fields()` (number/int → number input with
  step/min/max, bool → checkbox, axis → X/Y/Z select); one `capture('Edit Modifier')`
  per `change` commit. Empty states: "No active object" / "No modifiers". Refresh uses
  the signature-diff pattern (`id:modifiersVersion`) and defers rebuilds while a panel
  field holds focus, so it never yanks focus mid-edit.
- `src/io/sceneJson.ts` + test — format **v2**: writes `version: 2` and a per-object
  `modifiers: [{ type, name, enabled, params }]` array (params keys sorted, numbers
  rounded → deterministic). Loader accepts v1 (absent modifiers → `[]`) AND v2; unknown
  modifier types throw a readable error during validation, before any scene mutation;
  modifiers are rebuilt via `createModifier` up front (no half-load).
- `e2e/ship.mjs` — appended P4-4 checks: Modifiers tab + dropdown render, add/undo and
  param-edit/undo probes gated on `modifierTypes()` being non-empty (currently empty
  until P4-5, so those log SKIP and the empty-dropdown + "No modifiers" hint are
  asserted instead), plus format-v2 serialization assertions.

Verification: `npm run build` (tsc + vite) clean; `npm test` 137/137 (11 in
sceneJson.test.ts, incl. a 2-modifier round-trip using a test-local `registerModifier`,
v1-acceptance, and unknown-type rejection); all four e2e suites green
(smoke, edit, ship, workspace) run sequentially.

Deviations: none. The add/param undo e2e probes are written but SKIP at runtime because
no concrete modifiers are registered yet (P4-5) — exactly the gate the spec's acceptance
criterion 2 calls for.

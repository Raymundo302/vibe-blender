# UR6-1 — Line quality: proximity-scaled wire thickness, anti-aliased wires, fatter/smoother selection outline

## Context
The canvas context already has MSAA (`src/render/gl/context.ts` antialias:true);
the jaggedness Ray sees is inherent to 1px `gl.LINES`. The app already owns the
cure: `src/render/passes/intersectPass.ts` renders screen-space RIBBON quads with
a constant pixel width and soft alpha rims. This task brings that technique to
the mesh wires and thickens the selection outline.

## A. Wire lines → anti-aliased ribbons with proximity thickness
Convert the mesh-edge line rendering to ribbon quads:
- Applies to: wireframe shading mode, the Wireframe overlay, hidden-line mode,
  AND the edit cage's edge lines (`editOverlayPass`) — all currently gl.LINES.
  (Vert points, loop-cut preview, guides, grid: untouched.)
- **Width by view proximity**: per-endpoint pixel half-width
  `clamp(BASE_PX * REF_DIST / viewDist, MIN_PX, MAX_PX)` with
  BASE_PX 1.1, REF_DIST = the camera's orbit distance (so the look stays stable
  as you zoom — closer EDGES relative to the view are thicker), MIN 0.6, MAX 3.5.
  Constants in one place with a comment; expect eyes-on tuning by the architect.
- **AA**: soft alpha falloff over the outer ~1px of the ribbon (intersectPass
  rim pattern), blended, depth-write off for the soft rim (or single-quad alpha
  falloff — match intersectPass's proven approach).
- **Implementation latitude** (pick one, justify in your report):
  1. Instanced rendering: one instance per edge (divisor-1 attributes: endpoints
     A/B + the two adjacent face normals already in the edges VAO), corner index
     from a static 6-vert buffer, expand in the vertex shader; or
  2. CPU ribbon stream like segmentsToRibbon, cached per (gpuMesh version).
- MUST preserve existing behaviors exactly: hideBack orientation culling (both
  face normals back → cull), the hidden-line depth prime + fractional depth
  bias, the UR5-1 edit-object skip and see-through rules (UR5-1 is in your tree
  — read what it did to wirePass first), per-edge colors in the edit cage
  (orange selection), and the wireframe-mode theme color.
- Edit-cage edge SELECTION color must ride the ribbon (color per endpoint).

## B. Selection outline: thicker + smoother
`src/render/passes/outlinePass.ts`: today an averaged 8-tap circular kernel at
width w. Make the outline ~1.5× thicker and smoother:
- Double-ring kernel (16 taps: 8 at r, 8 at 0.6r, angularly offset) with the
  averaged-coverage falloff kept; raise the base width so the visible outline is
  ~3px at DPR 1. Constants documented for tuning.
- No color/behavior change otherwise (active-vs-selected tints, cage suppression
  for the edit object).

wirePass.ts, editOverlayPass.ts, outlinePass.ts, Renderer.ts (call-site

## Acceptance criteria
- tsc clean; npm test green.
- e2e (extend `e2e/shading.mjs` + `e2e/edit.mjs`): (1) proximity: with the cube
  filling the frame, a NEAR vertical edge's wire is measurably wider (count dark
  pixels across a perpendicular strip) than a FAR edge's at the same screen
  region scale — assert width_near > width_far + 1; (2) AA: a wire crossing a
  probe strip produces ≥2 intermediate color levels between wire core and
  background (not a binary edge); (3) selected cage edge is orange in the ribbon
  path (UR5-1 regression pinned); (4) outline: selected object's outline row
  count at a probe column is ≥3px and contains intermediate alpha levels;
  (5) hidden-line + wireframe suites, intersections overlay checks, and the
  banding/grazing AO checks ALL stay green.
- **Green on SwiftShader AND E2E_GPU=1** — this is precisely the
  driver-sensitive class (AO-saga lesson). Screenshots: near/far thickness,
  wireframe close-up, selection outline before/after — the architect does
  eyes-on.

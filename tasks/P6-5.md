# P6-5 — Proportional editing (O)

Read `tasks/P4-CONVENTIONS.md` first.

## Goal
Blender's proportional editing for edit-mode G/R/S: with **O** toggled on,
transforming a selection also moves nearby UNSELECTED verts with smooth
falloff. Mouse wheel DURING the modal adjusts the radius; a circle overlay
shows it at the pivot.

## Files
- `src/tools/editTransform.ts` — the only tool file to touch: weight map
  computed at start (weight = smooth falloff `3t²−2t³` of distance to the
  NEAREST selected vert, t = 1 − d/radius, clamped; selected verts weight 1);
  apply deltas scaled by weight. Radius lives in a module-level setting object
  exported for the overlay + persistence across invocations (default 2).
- `src/input/InputManager.ts` — O key toggles (status message), wheel while a
  proportional modal is active adjusts radius (guard: only when the active op
  is an edit transform with proportional on — add a narrow hook, do NOT break
  camera zoom otherwise).
- `src/render/Renderer.ts` — reuse `editPreviewLines` to draw the radius circle
  (48-segment polyline in the plane facing the camera at the pivot) while the
  modal runs — set/cleared by the operator like loop cut does. If loop-cut-style
  reuse conflicts, STOP and report.
- `e2e/edit.mjs` appended.

## Acceptance criteria
1. Unit (add to `src/tools/editTransform.test.ts`): falloff weights (d=0 → 1,
   d=radius → 0, d=radius/2 → 0.5, beyond → 0); a G-delta applied with
   proportional moves an unselected vert by weight×delta.
2. e2e: edit mode, select ONE corner vert, O on, G, move, confirm → the
   diagonally-opposite vert moved 0 but an adjacent vert moved a nonzero amount
   less than the selected one; Ctrl+Z restores; O off → adjacent vert stays put.
3. All four e2e suites + npm run build + npm test pass.

## Result
Built. Proportional editing (O) added to edit-mode G/R/S with a smooth 3t²−2t³
falloff of local distance to the nearest selected vert.

- `src/tools/editTransform.ts` — exported `proportional` settings singleton
  (`{ enabled, radius }`, default radius 2), `proportionalFalloff(d, radius)`,
  and `computeProportionalWeights(mesh, selected, enabled, radius)`.
  `EditTransformBase` now builds a per-vert weight map at start (all verts when
  proportional is on, selection-only when off), pivots on the SELECTED verts,
  applies deltas scaled by weight (translate) or blends world targets by weight
  (rotate/scale via `setWorld`), and exposes `adjustRadius(ctx, deltaY)` which
  the wheel calls to grow/shrink the radius, recompute weights, redraw the circle
  and re-apply live. The 48-segment influence circle is drawn in the pivot plane
  facing the camera through the reused `renderer.editPreviewLines` channel (local
  space), set on start / cleared on confirm+cancel — same pattern as loop cut.
  Renderer is an optional constructor arg so unit tests drive the op headless.
- `src/input/InputManager.ts` — `O` toggles `proportional.enabled` in edit mode
  with a status message; `onWheel` routes to `adjustRadius` only when the active
  op is an `EditTransformBase` with proportional active, otherwise camera zoom is
  untouched; the three edit operators are constructed with `this.renderer`.
- Tests: `editTransform.test.ts` covers the falloff values (d=0→1, d=r→0,
  d=r/2→0.5, beyond→0), the weight map (selected=1, neighbour falloff, far vert
  dropped, disabled=selection-only), and a G-delta moving an unselected neighbour
  by weight×delta while the opposite corner stays put. `e2e/edit.mjs` drives the
  full flow: select one corner, O on, G, wheel up the radius (read back from the
  status line), confirm → opposite vert unmoved, neighbour moved less than the
  selected vert, Ctrl+Z restores, O off → neighbour stays put.

Verification: `npm run build` clean, `npm test` 202 passed (24 files), all four
e2e suites green (smoke, edit, ship, workspace). No deviations; "weight map
computed at start" is honoured and additionally recomputed on wheel radius change
so the modal deforms live (required for the wheel to be meaningful mid-op).

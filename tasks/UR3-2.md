# UR3-2 — Intersect tool: embed mesh–mesh intersection curves into the meshes

## Goal
The "Intersections" shading overlay (src/core/mesh/intersect.ts + intersectPass)
only DRAWS where meshes pass through each other. New tool: with 2+ mesh objects
selected in object mode, running **Intersect** makes those crossings REAL geometry —
new verts on the edges that pierce the other mesh, and new edges splitting the
crossed faces — on EVERY selected mesh, in one undo step. Objects stay separate
(this is NOT a boolean union).

## Files
- NEW `src/core/mesh/ops/embedIntersections.ts` + `embedIntersections.test.ts` —
  pure geometry op (no GL/DOM), house style of `ops/knife.ts`.
- NEW `src/tools/intersectTool.ts` — the user-facing action:
  `export function runIntersectTool(scene: Scene, undo: UndoStack, setStatus: (s:string)=>void): void`
  (adjust param shape to what main.ts/toolbar can actually provide — keep it free
  of InputManager internals).
- Possibly small exports from `ops/knife.ts` if you reuse its edge-split /
  chord-split internals — prefer reuse over re-implementation, but do NOT change
  knife's behavior (its tests must stay green untouched).

## Algorithm (v1 — edge-piercing based, mirrors knife's machinery)
For each unordered pair (A, B) of the selected mesh objects:
1. Build world-space triangle data for each mesh from its BASE mesh (obj.mesh,
   NOT the modifier-evaluated mesh — we are editing real topology) transformed by
   the object's WORLD matrix (parent-aware — use the same world-matrix accessor
   the renderer/join use).
2. **Edge piercings:** for every edge of A, intersect the world-space segment
   against B's triangles (segment–triangle; write a small robust helper, or reuse
   any existing one — grep `closestPoint.ts` / raycast code first). Each hit at
   parameter t ∈ (0,1) → split that edge of A at t (canonical shared vert, exactly
   like knife: an edge shared by two faces gets ONE new vert). Same for B's edges
   against A's triangles. Dedupe hits closer than 1e-6 of an existing hit/vert on
   the same edge.
3. **Face splits:** any face that received exactly TWO new verts on its boundary
   is chord-split between them (same rule + winding preservation as knife.ts).
   Faces with 1 or >2 new verts just keep the boundary verts (manifold, no split)
   — document this exactly like knife does.
4. New verts get positions in the mesh's LOCAL space (inverse world matrix).

Return counts `{ verts: number; splits: number }` per mesh for the status line.

## Known v1 limitations — DOCUMENT in the op header, do not try to solve
- A face crossed only through its INTERIOR (no edge of that face pierces the other
  mesh — e.g. a large plane through a small cube: the plane's face) gains nothing.
  The cube in that scene DOES gain the full loop. Workaround for users: subdivide
  first. (Interior-loop embedding needs face-with-hole topology we don't have.)
- The chord is straight; on non-planar faces it approximates the true curve.
- UVs on modified faces follow knife's convention (dropped on modified faces).
- Self-intersection of a single mesh: out of scope (matches the overlay).

## Undo
One undo entry for the whole run: a command class (in
`src/core/undo/meshCommands.ts` or a new file) that composes per-mesh
`MeshEditCommand.capture`-style snapshots for every touched mesh — look at
`joinCommand.ts` / `separateCommand.ts` for multi-object command precedent.
Name: "Intersect".

## UX rules (in intersectTool.ts)
- Object mode only; needs ≥2 selected objects whose `kind` is mesh (see how join
  guards non-mesh). Otherwise `setStatus('Intersect: select 2+ mesh objects')`,
  no undo push.
- If zero segments found: status "Intersect: no intersections", no undo push.
- On success: status like `Intersect: 12 verts, 8 face splits across 2 objects`,
  meshes `touch()`ed / versions bumped the same way other mesh ops trigger GPU
  re-upload (grep how knife's confirm does it).
- Do NOT wire any keyboard shortcut and do NOT touch toolbar/main.ts — UR3-1
  imports `runIntersectTool` and wires the button.

## Acceptance criteria
- `npx tsc --noEmit` clean; `npm test` green (new unit tests included):
  - unit-cube edges pierced by an offset plane object → expected vert count on the
    4 pierced edge... (compute the real expectation: a plane crossing a cube
    horizontally pierces the 4 vertical edges → 4 new verts, 4 side-face splits).
  - transforms respected: same scene but cube translated/rotated via its object
    transform.
  - two-object symmetry: both meshes get edited when both have pierced edges
    (e.g. two interpenetrating cubes).
  - no-intersection → zero mutations, mesh version unchanged.
  - undo restores both meshes exactly (vert counts + positions).
- e2e: NEW small suite `e2e/intersect-tool.mjs`: build the plane-through-cube
  scene via `__app` handles, expose the tool for the test (export it on `__app`
  in intersectTool.ts is NOT possible without main.ts — instead have the test
  drive it once UR3-1 lands? NO: keep e2e in THIS task by calling the function
  through a module-level `window.__intersectTool` hook set inside
  intersectTool.ts itself, documented as an e2e handle like `window.__timeline`).
  Verify cube vert count grew by 4 and undo restores it. Unique `E2E_PORT`.
- If the spec is ambiguous or wrong (especially around knife-internal reuse),
  stop and report rather than improvising.

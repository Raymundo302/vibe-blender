# UR3-1 — Viewport Tool Palette (left-edge toolbar, mode-aware)

## Goal
Blender's T-toolbar: a vertical strip of tool buttons docked on the LEFT edge of the
3D Viewport, showing object-mode tools in object mode and mesh-edit tools in edit
mode. Clicking a button does exactly what pressing the tool's shortcut does.

## Files
- NEW `src/ui/toolbar.ts` + `src/ui/toolbar.css` — the widget + styling.
- EDIT `src/main.ts` — construct the Toolbar after the InputManager (it needs the
  InputManager instance), mount it, and call `toolbar.update()` in the existing
  frame loop (same place `topbar.update()` is called, ~line 425).
- Import `runIntersectTool` from `src/tools/intersectTool.ts` (built by UR3-2 —
  it EXISTS already when you start; do not stub it).

## Design
- Mounted inside `#viewport-wrap` (like the N-panel and splash), absolutely
  positioned: `left: 8px; top: 50%; transform: translateY(-50%)`, a slim vertical
  column (~36px wide buttons), themed like the existing dark panels (reuse the CSS
  variables from `src/ui/theme.css` / themes system — look at `nPanel.ts` +
  `shadingMenu.css` for the house style). Must NOT intercept viewport
  drags outside its own buttons (pointer-events only on the strip itself).
- Each button: an icon (use compact text/emoji glyphs consistent with the app's
  existing use, e.g. ↖ ✥ ⟳ ⤢ — pick tasteful ones) + a `title` tooltip
  "Name (Shortcut)". Hover highlight; a button whose operator is currently the
  active modal op gets an "active" highlight class (poll in `update()` — the
  InputManager needs a public readonly accessor for the active operator name; add
  a small `get activeOperatorName(): string | null` to InputManager).
- **Tool registry** (exported, typed): 
  `interface ToolDef { id: string; label: string; icon: string; shortcut: string; modes: ('object'|'edit')[]; run(): void }`
  built in toolbar.ts from a list. Buttons call `run()`.
- Mode-awareness: `update()` (frame loop) compares `scene.mode` (`'object' | 'edit'`,
  see `Scene.mode` getter) to the last-built mode and rebuilds the button list on
  change. Cheap when unchanged.

## Tools to register
Object mode:
- Move (G) → `inputManager.startOperator(new TranslateOperator())`
- Rotate (R) → RotateOperator
- Scale (S) → ScaleOperator
- Duplicate (Shift+D) — trigger the SAME code path the key uses. If that logic is
  inline in InputManager's key handler, extract it to a public method
  (`inputManager.duplicateSelected()`) and call it from both places. Do not copy
  the logic.
- Intersect (no shortcut) → `runIntersectTool(...)` from UR3-2. Object mode only.

Edit mode:
- Move / Rotate / Scale (edit variants — check which operator classes the edit-mode
  G/R/S branch constructs in InputManager (~line 1296) and use the same ones)
- Extrude (E), Inset (I), Bevel (Ctrl+B), Loop Cut (Ctrl+R), Knife (K),
  Edge Slide (GG) — each via the same operator/startOperator path the keyboard
  uses; extract shared helpers rather than duplicating any setup logic.

If an operator's `start()` returns false (e.g. no selection), it simply doesn't
start — the existing behavior; the button needs no special casing, but DO set a
status message via the same `setStatus` the ctx provides when nothing happens is
already handled by the op. No extra work.

## Out of scope
- No Blender-style persistent "active tool" that changes drag behavior — buttons
  are one-shot starters (v1).
- No icon artwork/SVG set.
- Don't touch the gizmo, sculpt tools, or the shading dropdown.
